# design/50-ui — model, update, render

> Spec: [spec/40](../spec/40-rendering.md), [spec/50](../spec/50-keybindings.md)

## What's in the package

`internal/ui` owns the picker's state machine and renderer. It is a
pure layer — no goroutines, no I/O, no time.Now() calls. The model
takes events in, produces a render Frame out. The fx graph wires
this layer to a real TTY in `internal/app`.

We did **not** use bubbletea. The package was sketched against
bubbletea early on (see the older drafts of this doc), but two
specific friction points pushed us to a hand-rolled FSM:

1. Bubbletea's built-in key parser splits bracketed-paste payloads
   across keystrokes; we need the payload as one event.
2. The picker's render is throttled per-write, not per-`tea.Msg`.
   Bubbletea's render lifecycle did not match this cleanly.

The resulting code is ~250 lines of pure Go in `model.go`,
`update.go`, and `render.go`; a fraction of the bubbletea shape
without losing testability.

## Model

```go
package ui

type Model struct {
    // Geometry — captured at startup and updated on ResizeEvent.
    InitCol int // 1-indexed column of the prompt at picker start
    InitRow int // 1-indexed row    of the prompt at picker start
    Width   int // terminal cols
    Height  int // terminal rows

    // Data.
    Choices []string // immutable; reverse-deduped history
    Filter  []string // current AndFilter result; rotated in place by Up/Down
    Idx     int      // selection within Filter (0-indexed)
    Limit   int      // dynamic; set by Render(); capped at MaxLimit
    Input   string   // current typed input
    Cursor  int      // caret offset within Input

    MaxLimit int // default 15; can be set via --max-limit

    // Termination state, set by Update on Enter / Esc / Ctrl-C.
    Submitted bool
    Canceled  bool
    Result    string
}
```

The fields are exported so tests can build models in odd states
without going through the constructor.

## Update

```go
func (m *Model) Update(ev keys.Event) (terminate bool)
```

Pure function — given (Model, Event) it mutates the Model and
returns whether the session should terminate (Enter / Esc /
Ctrl-C). No side effects, no I/O, no goroutines.

The dispatch covers:

- `RuneEvent` and `PasteEvent`: append to Input, recompute Filter
  via `search.AndFilter(Choices, Tokenize(Input))`, reset Idx to 0.
- `KeyEvent{Backspace}`: shorten Input by one *rune* (NOT one byte
  — multi-byte UTF-8 like CJK / emoji must delete atomically),
  recompute.
- `KeyEvent{CtrlU}`: clear Input, recompute.
- `KeyEvent{CtrlW}`: delete the last word (rune-walk: strip trailing
  whitespace, then strip the run of non-whitespace before it).
  Matches zsh's default `backward-kill-word`.
- `KeyEvent{Up/Down}` (also `CtrlP/CtrlN`): scroll one entry; rotate
  the visible window in place if at the boundary (see `moveUp` /
  `moveDown` in `update.go`). Ctrl-P / Ctrl-N are alias bindings to
  match zsh emacs-keymap muscle memory.
- `KeyEvent{PageUp/Down}`: rotate the visible window by
  `max1(Limit)` (with a `>=1` floor for the pre-Render case where
  Limit is still 0).
- `KeyEvent{Home}`: full filter recompute, Idx=0.
- `KeyEvent{End}`: walk Filter from the back accumulating wrapped
  row counts until heightLimit is hit, rotate by that count, focus
  the last entry. See `scrollToEnd` in `update.go`.
- `KeyEvent{Enter}`: set Submitted, Result := SubmitResult().
- `KeyEvent{Esc}` / `KeyEvent{CtrlC}`: set Canceled,
  Result := SubmitResult(). Both routes go through `SubmitResult`
  so the Canceled branch is exercised by every cancel path.
- `ResizeEvent`: update Width/Height. The next Render recomputes
  Limit.
- Unhandled events (Tab, Delete, etc.): silently ignored.

## Render

```go
type Frame struct {
    Pre       string // bytes to emit BEFORE drawing — clears prior frame
    Body      string // the new visible list
    Post      string // cursor restoration to the input row
    Size      int    // body rows BELOW input row N (input wrap rows + choice rows)
    Limit     int    // entries that ended up visible
    CursorRow int    // 0-indexed row offset where Post left the cursor
}

func (m *Model) Render(opts RenderOptions) Frame
```

`RenderOptions{PrevSize, PrevCursorRow}` — the caller passes the
previous frame's Size and CursorRow so Pre can walk back to the input
row and erase exactly the right number of rows below. `Pre`, `Body`,
and `Post` are kept separate so individual stages can be asserted in
tests without parsing escape sequences.

### Wrap-aware bookkeeping

The picker draws as an *overlay* on the user's existing zsh prompt at
a captured `initCol` — not as a full-screen TUI. So when the input row
overflows the terminal width (a long argv from `LBUFFER`, or a long
filter typed live) the input itself wraps to additional rows. Frame
fields account for both kinds of wrap:

- `inputExtra = InputExtraRows(InitCol, CellWidth(Input), Width)` —
  rows the input occupies below row N (0 when the input fits).
- `cursorRow, cursorCol = InputCursorPosition(InitCol,
  CellWidth(Input[:Cursor]), Width)` — where Post should land the
  caret. Convention: cursor sits one cell to the right of the last
  typed char on the row that char actually landed on; cursorCol clamps
  to `Width` when it would otherwise be `Width+1` (the deferred-wrap
  state every common terminal renders for an exactly-filling row).
- `Frame.Size = inputExtra + choiceRows`.

`Body` walks Filter while a running `rows` counter stays under
`heightLimit = choiceHeightLimit(Height, inputExtra)` — i.e.,
`Height - 3 - inputExtra`, floored at 1. Each entry contributes
`WrappedRowCount(sanitizeChoiceForRender(entry), Width)` to that
running sum (see [spec/40](../spec/40-rendering.md) for the formula).
The walk stops when the next entry would overflow OR when `Limit`
reaches `MaxLimit`. Sanitization is folded into the walk because the
raw bytes of an entry like `cmd \x1b[2J` would understate cell width
(runewidth treats control bytes as 0 cells) — the body must compute
wrap rows on the same string it actually writes.

Token highlighting wraps every matched substring in
`\x1b[1;36m...\x1b[0m` (bold cyan). The matcher merges overlapping
spans so the user never sees two open-codes in a row.

Setting `NO_COLOR=<non-empty>` per the
[no-color.org](https://no-color.org) convention suppresses the
SGR escapes — both the `highlight()` wrapping and the
per-entry SGR reset. Filtering / matching are unaffected; only
the cosmetic highlight is removed.

`Pre` walks up `PrevCursorRow` lines (CursorPrevLine) to reach the
input start row, erases the input row tail with
`cursor.toCol(InitCol) + eraseLineEnd`, walks down `prevSize` lines
erasing each (`\r\n + eraseLine` per row), then walks back up
`prevSize` lines and resets the column for Body to draw on. The very
first frame skips both walks since `prevSize == 0 && prevCursorRow ==
0` — only the input-row erase is emitted.

`Post` walks up `(Size - cursorRow)` lines from the bottom of the
body to land on the cursor's wrap row, then `cursor.toCol(cursorCol)`
positions the caret exactly. Without the wrap-aware walk-up, a clamped
`CursorToCol(initCol + Cursor)` would land on the wrong row whenever
the input overflowed terminal width.

## Throttle

```go
type Throttle struct { … }
func NewThrottle(interval time.Duration) *Throttle
func (t *Throttle) Fire(now time.Time) bool
```

Leading-edge throttle with a 72 ms window (matches the legacy port).
The event loop in `internal/app/loop.go` calls `Fire(time.Now())`
before writing a frame: returns `true` if enough time has passed,
`false` otherwise. A trailing-edge flush fires after the last burst
event so the final state of a paste is always rendered, even if
the burst started inside an existing throttle window.

The throttle never affects `Update` — the FSM mutates the model
on every event. Throttling is purely about how often we hand a
Frame to `tty.Writer`.

## Tests

`internal/ui/{model,render,update,wrap,highlight,throttle}_test.go`
exercise each layer independently. The model tests run with
`t.Parallel()` and use a fixture in `sampleChoices()`.

We did NOT use golden-frame tests. The render output contains
ANSI escapes that depend on `Width`/`Height`/`InitCol` — pinning
them as opaque byte strings would make refactoring the renderer
painful for marginal value. Instead each test asserts the
specific property it cares about (escape sequence presence,
highlight wrapping, dynamic-limit math).

## Performance budget

Benchmarks live in `internal/ui/bench_test.go` and
`internal/search/bench_test.go`; run with `task test:bench`. On
Apple M5 Pro (the lower bound for "modern dev box"):

| Bench | At N | ns/op | allocs |
| --- | --- | --- | --- |
| `BenchmarkRender` | 100,000 entries | ~20 µs | 173 |
| `BenchmarkHighlight` | 5 tokens, 1 line | ~200 ns | 7 |
| `BenchmarkAndFilter` | 100,000 entries | ~1.5 ms | 1 |
| `BenchmarkReverseDedupeUnescape` | 100,000 entries | ~800 µs | ~260 |
| `BenchmarkTokenize` | 22-char input | ~130 ns | 2 |

The 72 ms throttle window means we have headroom for ~50× the
current Render cost before noticing keystroke lag. AndFilter
runs once per keystroke (full re-filter) and stays well under
the throttle window even at 100k entries. ReverseDedupeUnescape
runs once at startup; sub-millisecond.

If a bench regresses by >2× in a PR, treat it as a real change
that needs justification. The benches are deterministic and
allocate-stable; CI does not gate on the numbers, but a stark
regression should be visible in the PR description.
